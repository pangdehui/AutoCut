## 它现在能做什么？

虽然视频处理的核心功能还在开发中，但基础的用户系统和积分体系已经跑起来了：

### 👤 用户系统
- **邮箱注册**：输入邮箱 → 收验证码 → 设密码，三步搞定
- **两种登录方式**：可以用验证码登录，也可以用密码登录，随你喜好
- **OAuth 登录**：如果你用的是 Manus 平台，可以直接一键登录
- **个人中心**：查看和编辑自己的账户信息
- **角色区分**：普通用户和管理员两种角色，管理员有更多操作权限

### 💰 积分系统
- **积分账户**：每个用户有独立的积分钱包
- **三种服务费率**：
  - AI 视频分析：10 积分 / 分钟
  - 视频剪辑：15 积分 / 分钟
  - 字幕生成：8 积分 / 分钟
- **完整流水**：每一笔积分的收入和支出都有记录，谁在什么时候做了什么，清清楚楚
- **管理后台**：管理员可以给用户充值或扣除积分
- **新人福利**：注册就送 1000 积分，足够体验各项功能

### 🎨 前端界面
- 首页有 Hero 展示区和功能介绍
- 仪表盘可以看到积分余额、上传视频（支持拖拽、批量、格式校验、2GB 限制）、任务列表
- 亮色 / 暗色两种主题，晚上用也不刺眼
- 手机上也能正常浏览（响应式布局）

### 🤖 AI 视频分析
- **智能抽帧**：FFmpeg 自动从视频中提取 6 个关键帧，均匀分布
- **多模态分析**：AI 模型分析帧内容，生成场景描述、关键词、精彩片段
- **降级处理**：AI 不可用时自动切换为模拟分析，保证功能可用
- **结果展示**：场景分析、关键词标签云、精彩片段评分、内容分类与摘要

### ⚙️ 任务队列
- **状态机**：排队中 → 处理中 → 已完成 / 失败，完整生命周期管理
- **自动处理**：服务端自动轮询队列，逐个处理待办任务
- **进度追踪**：实时显示 0-100% 处理进度
- **可扩展**：支持注册自定义处理器，适配不同的任务类型（分析/剪辑/字幕）

### 📤 视频上传
- **拖拽上传**：支持拖拽到页面或点击选择文件，单次最多 10 个
- **格式校验**：MP4、MKV、MOV、AVI、WebM、FLV 等主流格式
- **大小限制**：单个文件最大 2GB，前端 + 后端双重校验
- **批量处理**：可一次性选择多个文件，每个文件独立上传和状态反馈
- **本地存储**：文件保存在服务器的 `uploads/` 目录

## 技术选型：用了哪些东西？🛠️

选型的时候主要考虑了几个点：文档要全、社区要活跃、别太重。

| 层面 | 技术 | 为什么选它 |
|---|---|---|
| 前端框架 | **React 19** | 最流行的前端框架，教程多，遇到问题好找答案 |
| 类型系统 | **TypeScript** | 虽然写起来比 JS 麻烦，但自动补全和报错太香了 |
| CSS | **Tailwind CSS 4** | 不用在 CSS 文件和组件之间来回切，直接在 HTML 里写样式 |
| 组件库 | **shadcn/ui** | 不是传统的 npm 包，组件代码直接拷进项目里，想怎么改就怎么改 |
| 路由 | **wouter** | 轻量级路由库，API 和 React Router 类似但体积小很多 |
| API 通信 | **tRPC** | 前后端共享类型定义，不用手写接口文档，前端直接调后端函数 |
| 后端 | **Express** | Node.js 最经典的框架，中间件生态丰富 |
| 数据库 ORM | **Drizzle ORM** | SQL-like 的查询语法，类型安全，比 Prisma 更轻量 |
| 数据库 | **MySQL / TiDB** | 经典关系型数据库，稳定可靠 |
| 认证 | **jose + JWT** | 轻量 JWT 库，不依赖 crypto 全局对象 |
| 数据校验 | **Zod** | 定义一次 Schema，前后端都能用 |
| 包管理 | **pnpm** | 比 npm 快，省磁盘空间 |
| 构建 | **Vite 7** | 开发时热更新飞快，构建也快 |

### 一些技术细节

**为什么用 tRPC 而不是 REST？**
tRPC 让我在 `server/routers.ts` 里定义一个函数，前端就能直接调用它，而且类型是自动推导的。不用写 axios 封装、不用手写接口类型、不用维护 API 文档——对一个人的项目来说，这省了太多事。

**为什么用 Drizzle ORM 而不是 Prisma？**
Drizzle 的查询语法更接近原生 SQL，学习成本低。而且它生成的 SQL 很干净，不会有 Prisma 那种嵌套 N 层的 JOIN 怪物。

**为什么用 wouter 而不是 React Router？**
wouter 的 API 几乎和 React Router 一样，但整个库只有 1KB 多。对于这个项目规模完全够用了。

### 改代码时应该动哪些文件？

- **想加新功能？** → `server/services/` 新建服务文件 → 在 `server/routers.ts` 注册路由 → 在 `client/src/pages/` 写页面
- **想改数据库？** → 修改 `drizzle/schema.ts` → 运行 `pnpm db:push` 生成迁移
- **想调 UI？** → 去 `client/src/components/` 找对应的组件
- **想加新页面？** → `client/src/pages/` 新建 → 在 `App.tsx` 里加一条 `<Route>`
- **_core/ 里的文件**→ 尽量不要动，那是框架模板代码

## 请求是怎么走的？从头到尾 🔄

可能有人会好奇，用户在浏览器里点一个按钮，数据是怎么到数据库又返回来的。这里画一个简化的流程图：

```
你在浏览器里点击「获取积分余额」
         │
         ▼
    tRPC 客户端发起请求 (useQuery)
         │
         ▼
  POST /api/trpc/credits.getBalance ──► Express 服务器
         │                                  │
         │                           解析 Cookie 里的 JWT
         │                                  │
         │                           验证签名、提取用户信息
         │                                  │
         │                           查数据库：userCredits 表
         │                            where userId = 当前用户
         │                                  │
         │                                  ▼
         │                           返回 { balance: 1000, ... }
         │                                  │
         ▼                                  ▼
   UI 自动更新，积分数字显示在页面上
```

### 三种接口权限

```typescript
// 不需要登录就能调
publicProcedure      // 注册、发送验证码

// 必须登录才能调
protectedProcedure   // 查积分、个人中心

// 必须是管理员才能调
adminProcedure       // 给用户充值、扣积分
```

## 数据库：9 张表是干什么的？📊

> 不用担心，刚开始看可能觉得表很多，其实每张表的用途都很明确。

### 用户相关（3 张表）

```
users                    用户的账号信息（邮箱、密码、角色等）
  │
  ├── emailVerificationCodes    注册/登录时的验证码（10 分钟过期）
  └── userCredits               积分钱包（余额、累计赚取、累计消耗）
```

### 积分相关（2 张表）

```
creditRates              费率表（分析/剪辑/字幕各多少钱一分钟）
creditTransactions        每一笔积分的流水账（谁、什么时候、消费还是充值）
```

### 视频相关（4 张表，⚠️ 表建了但功能没做）

```
videos                   上传的视频文件信息（路径、大小、时长）
processingTasks          视频处理任务（排队中→处理中→完成/失败）
videoAnalysis            AI 分析的详细结果（场景、关键词、精彩片段）
subtitles                字幕数据（多语言、SRT 格式）
```

## 怎么跑起来？🚀

### 第一步：准备工作

你需要电脑上装了这些：

- **Node.js** ≥ 18（[官网下载](https://nodejs.org)）
- **pnpm** ≥ 10（装好 Node.js 后，终端运行 `npm install -g pnpm`）
- **MySQL** ≥ 8.0（本地装一个，或者用 Docker）

### 第二步：克隆代码

```bash
git clone https://github.com/pangdehui/AutoCut.git
cd AutoCut
pnpm install
```

### 第三步：配置数据库

先在 MySQL 里创建一个数据库：

```sql
CREATE DATABASE autocut CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

然后在项目根目录创建 `.env` 文件（⚠️ 这个文件不提交到 Git）：

```bash
# 必填：数据库连接
DATABASE_URL=mysql://root:你的密码@localhost:3306/autocut

# 必填：JWT 密钥（随便打一串乱码就行，越长越安全）
JWT_SECRET=随便打一串至少32个字符的乱码abcdefghijklmnopqrstuvwxyz

# 选填：OAuth 登录（没有就不填，邮箱登录照样用）
VITE_APP_ID=
OAUTH_SERVER_URL=
OWNER_OPEN_ID=

# 选填：文件存储（没有就不填）
BUILT_IN_FORGE_API_URL=
BUILT_IN_FORGE_API_KEY=
```

### 第四步：初始化数据库

```bash
pnpm db:push
```

这个命令会自动根据 `drizzle/schema.ts` 里的表定义，在数据库里创建所有表。

### 第五步：启动！

```bash
pnpm dev
```

打开浏览器访问 **http://localhost:3000**，你应该能看到 AutoCut 的首页。

## 常见问题 FAQ ❓

**Q: 为什么不直接用 FFmpeg 命令而是要做成网站？**
A: 会用命令行的人毕竟是少数，我希望让不太懂技术的视频创作者也能用上 AI 剪辑。而且把 AI 分析和自动剪辑结合起来，比手动敲命令效率高很多。

**Q: 邮件发不出去怎么办？**
A: 现在的邮件服务只是一个模拟实现（代码里用 `console.log` 替代了发送）。如果你要部署到生产环境，需要去申请 SendGrid 或 Mailgun 的 API Key，然后替换 `server/services/emailService.ts` 里的 `sendVerificationEmail` 函数。

**Q: 为什么选 MySQL 而不是 PostgreSQL？**
A: 因为 TiDB 兼容 MySQL 协议，云上的 TiDB 免运维。如果你更喜欢 PostgreSQL，用 Drizzle 切过去也不麻烦。

**Q: 1000 积分能用多久？**
A: 按分析 10 积分/分钟算，可以分析 100 分钟的视频；按剪辑 15 积分/分钟算，可以处理约 67 分钟。作为体验来说足够了。

**Q: 前端为什么这么多组件文件？**
A: 因为用了 shadcn/ui，它不是传统的 npm 包，而是把组件源码直接放到你的项目里。好处是你可以随意修改组件内部代码，不用等上游更新。

## 已知问题（诚实地说）⚠️

我不希望给你一个"这个项目很完美"的错觉。以下是我已经知道但还没来得及修的问题：

1. **邮件是假的**：`server/services/emailService.ts` 里只打了 `console.log`，真用的话要接 SendGrid 或 Mailgun
2. **FFmpeg 剪辑和字幕还没做**：视频剪辑和字幕生成功能尚未实现，AI 分析已可用
3. **任务队列是内存级的**：当前队列在服务进程内运行，重启后队列清空。后续可升级为 Bull/RabbitMQ 实现持久化和分布式处理
4. **云存储依赖 Forge 平台**：`storage.ts` 里的 S3 上传逻辑依赖 Forge 代理，视频上传用的是本地存储，如果需要云存储（如 AWS S3、阿里云 OSS）需要自己适配
5. **测试几乎为零**：整个项目只有 1 个测试文件，测的是退出登录功能。正经项目应该有几十上百个测试
6. **后台管理是空的**：Dashboard 里的管理按钮点了没反应，对应的页面和 API 都还没写
7. **没有日志系统**：出了错只能看控制台，没有一个正经的日志收集方案

如果你在用的过程中发现了其他问题，请一定提 Issue 告诉我！
